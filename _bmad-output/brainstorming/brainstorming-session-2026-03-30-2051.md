# Brainstorming Session Results

**Facilitator:** Phoenix
**Date:** 2026-03-30

## Session Overview

**Topic:** 机器狗视觉与定位系统和运动控制之间的接口设计
**Goals:** 设计稳定、高效、可扩展的通信和数据接口

### Session Setup

针对机器狗的视觉与定位系统（Dog Vision & Localization）与下层运动控制系统之间的接口交互进行头脑风暴，探索接口的通信协议、数据结构及架构设计方案。

### Current Context
- 箱子位置信息：在起点解算，转换为相对于世界坐标系和机器狗坐标系的位姿，通过 ROS 2 话题 `/localization/boxes` 发布。
- 机器狗位姿信息：持续发送机器狗行进位姿，通过 ROS 2 话题 `/localization/dog` 发布。
- 机器狗任务：
  1. 从起点出发，识别并计算场地角落的数学题。
  2. 拾取场地中间8个位置的8个箱子（4种类型）。
  3. 分类放置在场地顶边回收区。
  *补充约束*：机器狗有4个爪子，计划分2次抓取（每次4个）。箱子摆放固定（2排4列），机器狗在两排中间一次性抓取4个。视觉侧维护一张“爪子-箱子种类”的映射表。

### Ideas Generated (Phase 1)
- **Idea 1 (Combine Response):** 瘦客户端执行模式（Thin Client）。运动控制系统仅作为纯粹的执行层，视觉定位系统掌握全局导航策略，因此无需向底层发送无用的全局或未来路径上下文。
- **Idea 2 (Modify Response):** 全量实时广播（Full Broadcast）。8个箱子的空间三维坐标和类型（即使加上协方差矩阵和ID）数据量极小（不到千字节级别），在 ROS 2 中即使以 30Hz 持续高频发布，网络和 CPU 开销也几乎可以忽略不计。直接长周期广播所有精确坐标反而简化了上层状态机同步。
- **Idea 3 (Eliminate Response):** 话题推送目标群（Topic-based Target Grouping）。废弃一问一答的 Service RPC 模式以避免绕远路和时间损耗。视觉系统直接通过话题发送当前目标批次（如一次性发送4个目标箱子的坐标），运控层自行计算这4个坐标的中心点并解算4个爪子的IK（逆运动学）进行并发抓取。

### Ideas Generated (Phase 2 - SIT)
- **Idea 4 (Task Unification/Dropoff Strategy):** 最优放置序列指令。视觉规划层根据内部维护的“爪子 ID - 箱子类型”映射表，自动计算出时间最短的最优放置顺序，然后直接向运动控制下发如“将 1号爪 的箱子放置到 坐标X”这种纯粹的动作执行指令。运控层无需进行任何逻辑判断。
- **Idea 5 (Separation of Concerns/Math Result):** 评分逻辑与运动控制完全解耦。数学题的计算结果只与最终得分和展示环节（例如通过屏幕或LED显示）有关，不影响“将所有箱子搬回”的全局最优调度路径。因此，该计算结果在视觉/规划侧内部闭环处理，绝不通过接口泄露或发送给运动控制。

### Edge Cases & Exception Handling (边界情况与异常处理)
- **环境约束明确：** 场地内仅有单支队伍，不存在其他队伍干扰造成的异常（如箱子被撞倒）。环境相对确定，主要误差来源于狗自身的执行偏差。
- **致命异常探讨：** 抓取落空（抓空）。需要设计反馈机制或冗余检测来处理“动作执行了但实际上没抓到”的情况。
- **抓空检测策略（视觉闭环验证）：** 因为狗不能随意抬头/低头以免影响相机位姿解算，系统将采用“视觉层自主验证”方案。运动控制层无需改动接口提供硬件反馈。系统在完成一次抓取后，直接通过实时的画面捕捉遗留的箱子（原本应该被抓起但仍留在视野/地面标定位置的箱子即可判定为未抓取成功），实现“可见即所得”的闭环检测，保持了极简的接口设定。
- **抓空补救策略（原位微调强制重试）：** 狗的硬件约束是严格的4爪并行，且总量限制在两趟满载（4+4=8），无法在第二次运行时强行挂载5个抓取任务。因此，当系统通过“视觉识别”判断出某个/某些爪子发生抓空时，系统必须立即进行位姿微调，并向运动控制重发局部精调抓取命令，直到该爪子被成功占据（满载）后方可触发“运送放置”流程。
