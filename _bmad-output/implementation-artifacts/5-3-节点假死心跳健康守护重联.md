# Story 5.3: 节点假死心跳健康守护重联

Status: done

## Story

As a 机器狗系统,
I want 一直侦听相机的出图率并在卡死时对其全生命周期的剔除并重启,
so that 根治底层驱动挂机（FR-3.1）。

## Acceptance Criteria

1. Given 死锁的传感器网络导致主节点 > 2s 无数据入站，When 心跳失败，Then 强制 Lifecycle 降为 Inactive 再回到 Active。
2. Given 执行重联拉起流程，When 节点恢复订阅并重新输出有效数据，Then 端到端恢复耗时 < 2 秒（NFR-3）。
3. Given 重联次数过多或持续失败，When 达到可配置阈值，Then 输出结构化告警并进入受控降级，禁止无限重启风暴。

### Technical Requirements

- 必须满足 FR-3.1：持续 2 秒无帧时执行健康自检报警并拉起死锁订阅节点。
- 必须满足 NFR-3：单节点重联后恢复有效输出总耗时 < 2 秒。
- 心跳判定、重启窗口、失败上限全部参数化（YAML）；禁止硬编码魔法数。
- 所有状态切换与失败路径统一使用 `RCLCPP_INFO/WARN/ERROR`，日志携带时间、节点名、原因码。

### Architecture Compliance

- `dog_lifecycle`：负责心跳监测、Lifecycle 状态迁移、失败阈值控制与告警。
- `dog_perception`：提供可判定“有效帧”的信号/时间戳，不承担重联编排。
- `dog_behavior`：消费降级状态并维持任务连续性，不重置上下文。
- 保持 `.hpp/.cpp` 分离与 ROS 2 生命周期节点模式（`rclcpp_lifecycle::LifecycleNode`）。

### Library & Framework Requirements

- 技术栈锁定 ROS 2 Humble + `rclcpp`/`rclcpp_lifecycle`；不引入新中间件。

- 回调链路保持零拷贝友好与固定上限容器策略，避免内存抖动。
