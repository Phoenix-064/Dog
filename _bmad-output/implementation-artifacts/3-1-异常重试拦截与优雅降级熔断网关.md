# Story 3.1: 异常重试拦截与优雅降级熔断网关

Status: done

## Story

As a 领队,
I want 机器狗在屡次抓空时果断舍去死锁目标直接降级,
so that 保障不因为卡点吃罚分（FR-3.2, FR-3.3）。

## Acceptance Criteria

1. Given 连续收到 2 次抓空，When 达到 YAML 定义阈值，Then 请求被熔断并抛退，系统必须在 < 1 秒内果断抛异常并切换至降级闭环状态（NFR-4）。

## Dev Notes

- Epic 3 目标是“异常熔断与任务降级”，本故事是该 Epic 的首个故事，负责建立最小可用的重试熔断网关。
- 与 Epic 2 的关系：复用 `dog_behavior` 已形成的动作反馈语义（成功/失败/拒绝/超时）作为守护输入，不重复实现 Action 通信层。
- 降级策略需遵循 PRD 中“保分优先，不死锁”的原则：触发熔断后优先推进可完成任务，而非无限重试当前目标。

### Technical Requirements

- 使用 ROS 2 Humble（`rclcpp`）实现守护节点，事件驱动优先，避免周期性重负载轮询。
- 阈值与时序参数外置到 YAML / 参数系统，不允许硬编码在 `.cpp` 中。
- 若需跨包事件消息，优先复用 `dog_interfaces` 现有契约；如确需新增消息定义，必须同步更新 `dog_interfaces` 的 `CMakeLists.txt` 与 `package.xml`。

### Architecture Compliance

- `dog_behavior` 继续作为动作通信边界，`dog_lifecycle` 负责守护判定与降级裁决；禁止反向耦合到感知包。
- 严格遵守“回调非阻塞”与“参数仅初始化期读取”规则。
- 命名与结构遵循工程规范：`.hpp/.cpp` 分离、成员变量后缀 `_`、Topic/Action 名称小写下划线。

### Library & Framework Requirements

- ROS 2 Humble：`rclcpp`、`std_msgs`（如复用）、`dog_interfaces`。
- 测试框架沿用 `ament_cmake_gtest` 与现有测试风格。
- 严禁引入与本故事无关的重型新依赖。
