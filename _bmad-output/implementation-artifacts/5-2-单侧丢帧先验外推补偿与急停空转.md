# Story 5.2: 单侧丢帧先验外推补偿与急停空转

Status: done

## Story

As a 机器狗系统,
I want 传感器突然失能时能以外推或空转应对,
so that 机器不会由于突兀的 0 输入彻底失控（FR-1.5, FR-3.5）。

## Acceptance Criteria

1. Given 一方流断联 > 150ms，When 队列出现空档，Then 从 `circular_buffer` 取出先验前推输出。
2. Given 接收物理急停，When 系统进入待命安全态，Then 转为降低 CPU 计算量的纯空转位姿发布而不重置任务状态。
3. Given 系统长期待机与实战切换，When 执行外推/空转策略，Then 满足连续 2 小时稳定运行且无 OOM（NFR-5）。

### Technical Requirements

- 必须遵循 FR-1.5（单侧短暂缺失外推）与 FR-3.5（急停低负荷空转）实现闭环。
- 丢帧判定和外推触发要参数化（YAML），默认窗口 `150ms`，禁止硬编码魔法数。
- 外推链路使用定长队列（`circular_buffer` 或等效固定上限实现），不得引入无界缓存。
- 任何降级路径必须非阻塞，禁止在回调内 `std::this_thread::sleep_for()` 或忙等循环。
- 日志统一 `RCLCPP_INFO/WARN/ERROR`，必须带上模式切换原因与关键时序信息。

### Architecture Compliance

- `dog_perception`：负责断流判定、历史先验外推与稳定发布。
- `dog_lifecycle`：负责 E-Stop/系统模式门控、降级状态编排与恢复回切协调。
- `dog_behavior`：消费降级状态并维持任务状态机连续性，不重置任务上下文。
- 保持 `.hpp/.cpp` 分离、包前缀 include 规范、高频路径参数初始化期读取。

### Library & Framework Requirements

- 技术栈锁定：ROS 2 Humble + `rclcpp`，沿用现有消息契约与构建系统。
- 使用项目既有 `yaml-cpp` 配置机制管理阈值与模式参数，不新增数据库或外部状态服务。
- 队列实现遵循架构约束：固定上限容器优先，避免动态扩容导致内存抖动。
