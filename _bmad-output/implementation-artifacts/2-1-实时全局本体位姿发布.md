# Story 2.1: 实时全局本体位姿发布

Status: in-progress

## Story

As a 外包运控系统,
I want 实时获取狗的地图级位姿以作逆运动学解算,
so that 可以指引机器狗正确运动至抓取区。

## Acceptance Criteria

1. Given 感知里程计，When 狗体移动，Then 输出 `PoseStamped` 给运控，网络延时波动 < 5ms（NFR-6）。

## Dev Notes

- 本故事对应 Epic 2 的入口故事，目标是先打通“运控可消费的本体位姿输出”最小闭环。
- 当前代码中 `BehaviorNode` 已具备里程计订阅骨架（`localization_topic`）与占位发布逻辑，必须优先复用，不重建并行节点。
- 设计边界：Story 2.1 仅交付 `PoseStamped` 实时发布；动作指令下发与异步反馈监听在 Story 2.2/2.3。

### Technical Requirements

- 消息契约：对外发布类型为 `geometry_msgs::msg::PoseStamped`。
- 数据来源：来自 `nav_msgs::msg::Odometry` 感知里程计；转换时保持时戳与坐标系一致性。
- 性能约束：满足 NFR-6（通信延时波动 < 5ms），避免高频链路中的阻塞与重复分配。
- 参数策略：所有 topic / 默认 frame 参数在节点初始化期声明并读取，不在回调中查参。

### Architecture Compliance

- 继续使用 `dog_behavior` 作为规划/对外通信边界，不在 `dog_perception` 或 `dog_lifecycle` 新增重复发布职责。
- 遵循 `.hpp/.cpp` 分离、成员后缀 `_`、ROS 命名小写下划线、日志宏规范。
- 回调实现保持非阻塞；不得在回调中执行 `std::this_thread::sleep_for`。
- 若后续需要多路同步（图像+里程计），必须采用 `message_filters`；本故事不提前引入无关复杂度。

### Library & Framework Requirements

- ROS 2 Humble（`rclcpp`）+
  `geometry_msgs` / `nav_msgs` 标准消息。
- QoS 依据架构基线：高频订阅优先 `SensorDataQoS`，关键通信发布使用系统默认可靠策略。
- 内存管理遵循智能指针与零拷贝优先原则，不引入裸指针所有权风险。
