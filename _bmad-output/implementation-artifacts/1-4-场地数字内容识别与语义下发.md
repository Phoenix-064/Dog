# Story 1.4: 场地数字内容识别与语义下发

Status: done

## Story

As a 机器狗系统,
I want 能够识别截取场地中的数字标识并输出结构化语义结果,
so that 行为规划层可以据此规划路径偏好与后续动作策略。

## Acceptance Criteria

1. Given 感知图像中存在特定数字区，When 独立识别流执行，Then 发布带数字类型的消息流，耗时 < 50ms（NFR-1）。
2. And 在强光干扰下，识别框伪阳性错报率 < 2%（NFR-7）。

## Dev Notes

- 本故事目标是补齐 Epic 1 中 FR-1.4：完成“数字题识别 -> 语义结果发布”的独立能力，不引入运控动作下发逻辑（运控交互归 Epic 2）。
- 必须延续 Story 1.3 已落地的数据链路防线：零拷贝、固定容量缓存、参数初始化期读取、回调非阻塞。
- 识别模块需保持可替换：算法实现可替换，Node 框架稳定，支持比赛现场快速调参与替换。

### Technical Requirements

- ROS 2 Humble，C++17，`rclcpp` + `sensor_msgs` + `message_filters` + `tf2`/`tf2_ros`。
- 识别算法层使用抽象接口 + 工厂返回 `std::unique_ptr<Interface>`；ROS Node 不使用工厂。
- 回调路径禁止 `std::this_thread::sleep_for()`，禁止高频回调内 `get_parameter()` 和磁盘 I/O。
- 所有日志统一 `RCLCPP_INFO/WARN/ERROR`，禁止 `std::cout/printf`。

### Architecture Compliance

- 目录与职责遵循四包边界：`dog_interfaces`（契约）、`dog_perception`（感知解算）、`dog_behavior`（规划外发）、`dog_lifecycle`（守护恢复）。
- 感知算法类与状态管理层分离：算法只处理单帧推理，状态平滑由独立缓存/包装层承担。
- 统一遵循“错误时宁可无特征，不可错特征”原则，防止错误语义进入行为层。

### Library & Framework Requirements

- 必要依赖：`rclcpp`、`sensor_msgs`、`message_filters`、`tf2`/`tf2_ros`、`Boost (circular_buffer)`。
- 若新增 OpenCV 组件或其他库，必须同步更新 `package.xml` 与 `CMakeLists.txt`。
- QoS 需与发布端兼容（传感器流优先 `SensorDataQoS` / best-effort 场景），避免“无警告不回调”。
