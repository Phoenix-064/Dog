# Story 1.5: 持久化基础设施准备与 IO 工具链

Status: done

## Story

As a 视觉底层节点运维,
I want 预建立基于 `yaml-cpp` 的文件流操作服务模块（供其它 Epic 调用）, 
so that 异常状态和任务追踪可以有安全的介质存取。

## Acceptance Criteria

1. Given 存储介质，When 任意节点调用持久化接口，Then 文件能够安全执行同步小包快速刷写（Flush）。

## Dev Notes

- 本故事是 Epic 1 的“基础设施收口”项，核心是把状态持久化能力做成可复用底座，直接服务 Epic 5.1 的断电恢复。
- 当前 Epic 1 中 Story 1.3 仍处于 `in-progress`，实现时避免引入与感知主链路耦合的高频磁盘访问。
- 重点不是“存很多”，而是“写得稳、读得准、坏了可恢复、接口可复用”。

### Technical Requirements

- 技术栈：ROS 2 Humble + C++17 + `yaml-cpp`（ROS 2 生态中 `yaml_cpp_vendor` 维护，Humble 为推荐使用）。
- 参数规则：遵循 ROS 2 参数模型，参数与节点生命周期绑定；参数应在初始化阶段声明并加载，避免高频路径频繁查询。
- IO 规则：禁止在高频 Topic 回调中进行阻塞式文件写；持久化触发应由状态机事件驱动。
- 代码规则：统一 `RCLCPP_*` 日志，禁止 `std::cout/printf`；新增依赖需同步更新 `CMakeLists.txt` 与 `package.xml`。

### Architecture Compliance

- 职责边界必须保持：`dog_lifecycle` 负责状态持久化，`dog_behavior`/`dog_perception` 仅消费服务能力。
- 遵循 `.hpp/.cpp` 分离、库化封装（`add_library`）与路径规范（全路径 include）。
- 需兼容既有“正常结束删除状态文件、异常中断保留状态文件用于恢复”的架构决策。

### Library & Framework Requirements

- `yaml-cpp` 解析/序列化：使用 `YAML::LoadFile`、`YAML::Node` 与类型安全 `as<T>()` 读取。
- 写入策略要求显式 flush，并确保崩溃场景不会产生不可解析中间态。
- 如引入文件系统工具（例如 `std::filesystem`）或同步机制，必须评估编译标准与平台兼容。

### File Structure Requirements

- 主要改动预期：
  - `src/dog_lifecycle/include/dog_lifecycle/`（新增持久化接口与实现头）
  - `src/dog_lifecycle/src/`（新增 YAML 持久化实现与接线逻辑）
  - `src/dog_lifecycle/config/`（新增持久化配置，如路径、刷新策略、版本号）
  - `src/dog_lifecycle/test/`（新增持久化测试）
  - `src/dog_lifecycle/CMakeLists.txt`
  - `src/dog_lifecycle/package.xml`

### Testing Requirements

- 功能正确性：写入后可完整读回；字段缺失/类型错误能被拦截并返回可诊断结果。
- 稳定性：反复写入不出现资源泄漏或句柄泄漏；损坏文件可进入可控降级。
- 性能约束：小包写入满足“快速刷写”目标，不阻塞关键实时链路。
- 工程质量：`-Wall -Wextra` 下无新增 warning，相关包测试通过。
