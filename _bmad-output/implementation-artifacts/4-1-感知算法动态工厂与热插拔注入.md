## Dev Notes

- Epic 4 目标是“赛场敏捷调试与热重载架构”，本故事是该 Epic 的入口故事，核心在于“算法实现可替换、节点主逻辑稳定”。
- 需求重点不是“在线热更新共享库”，而是通过工厂边界实现低成本替换与快速重编部署，避免牵连主干。
- 必须优先复用现有目录与构建组织，不引入与目标无关的新框架。

### Technical Requirements

- 工厂边界：算法类必须通过抽象接口 + 工厂创建，返回 `std::unique_ptr`；节点类本身不强制使用工厂。
- 内存与回调：遵循零拷贝/智能指针策略，避免在高频回调中发生大对象复制。
- 参数策略：算法类型、阈值等由 YAML/参数系统注入，仅在初始化期读取。
- 日志策略：统一使用 `RCLCPP_INFO/WARN/ERROR`，禁止 `std::cout`。

### Architecture Compliance

- 保持四包边界：仅在 `dog_perception` 内实现算法可替换，不跨包侵入 `dog_behavior`/`dog_lifecycle`。
- 保持 `.hpp/.cpp` 分离与 include 路径规范（带包名前缀）。
- 禁止在回调链路中引入 `sleep_for`、忙等或阻塞式等待。

### Library & Framework Requirements

- ROS 2 Humble + `rclcpp`，遵循既有 `ament_cmake` 与包依赖声明。
- 感知相关继续沿用 OpenCV/PCL 已有能力，不新增重型依赖（除非确有必要并同步更新 `package.xml`、`CMakeLists.txt`）。
- 推荐采用 `message_filters` 等现有机制维持数据对齐策略，不单独引入新同步框架。

### File Structure Requirements

- 预期主要改动位置（以最小改动为准）：
  - `src/dog_perception/include/dog_perception/detector_base.hpp`
  - `src/dog_perception/include/dog_perception/detector_factory.hpp`
  - `src/dog_perception/src/detectors/*.cpp`
  - `src/dog_perception/src/*node*.cpp`
  - `src/dog_perception/CMakeLists.txt`（按需）
  - `src/dog_perception/package.xml`（按需）
  - `src/dog_perception/test/*`（按需）

### Testing Requirements

- 工厂行为测试：给定合法算法类型返回正确实现；非法类型返回可观测错误并安全回退。
- 契约不变测试：替换实现后，输出消息类型与关键字段契约保持一致。
- 构建质量测试：`-Wall -Wextra` 下无新增 warning，且 `dog_perception` 定向测试通过。

### Latest Tech Information

- 当前项目基线明确为 ROS 2 Humble 与现代 C++ 实践；本故事应保持与该基线一致，不做版本漂移。
- 对“热插拔”采用工程可落地解释：通过抽象工厂最小改动替换实现并快速重编，不在本故事引入复杂运行时插件系统。
